# 時刻同期アルゴリズム

本ドキュメントでは、本アプリケーションにおける NTP 時刻同期の仕組みを説明する。

## 概要

クライアント（ブラウザ）の `Date.now()` はOSの時計に依存しており、数百ミリ秒〜数秒のずれが生じ得る。本アプリケーションでは、サーバーが NTP サーバーから取得した正確な時刻情報を中継し、クライアントが **自身のPC時刻を基準** に NTP オフセットを算出・補正する。

## アーキテクチャ

```
┌──────────┐     UDP/123      ┌──────────────┐     HTTP      ┌──────────────┐
│ NTP サーバー │ ◄──────────► │  Web サーバー   │ ◄──────────► │  ブラウザ      │
│ ntp.nict.jp│               │  (Node.js)    │              │ (クライアント) │
└──────────┘               └──────────────┘              └──────────────┘
```

## Phase 1: サーバー ↔ NTP サーバー

### 処理フロー

サーバーは起動時および **60秒間隔** で NTP サーバー (`ntp.nict.jp`, ポート123) に問い合わせを行う。

```
T_send  = サーバーが NTP リクエストを送信した時刻 (Date.now())
T_recv  = サーバーが NTP レスポンスを受信した時刻 (Date.now())
T_ntp   = NTP サーバーから返却された時刻 (ms)
```

### オフセット計算

```
RTT_server = T_recv − T_send
T_local_mid = T_send + RTT_server / 2
serverOffset = T_ntp − T_local_mid
```

- `RTT_server`: サーバーと NTP サーバー間の往復時間
- `T_local_mid`: リクエストの中間時点におけるサーバーのローカル時刻（NTP レスポンスがこの瞬間の時刻を示すと仮定）
- `serverOffset`: サーバーのローカル時計に対する NTP 時刻のずれ（ミリ秒）。正の値はサーバーの時計が遅れていることを示す

### 保持データ

| 変数          | 内容                                      |
| ------------- | ----------------------------------------- |
| `ntpOffset`   | `serverOffset` の最新値 (ms)              |
| `ntpLastSync` | 最後に同期成功した時刻 (ISO 8601)         |
| `ntpError`    | 最後のエラーメッセージ（成功時は `null`） |

## Phase 2: サーバー API

### エンドポイント

```
GET /api/ntp-offset
```

### レスポンス

```json
{
  "offsetMs": 12.5,
  "lastSync": "2026-04-13T12:00:00.000Z",
  "server": "ntp.nict.jp",
  "error": null,
  "serverTime": 1776182400000
}
```

| フィールド   | 型               | 説明                                              |
| ------------ | ---------------- | ------------------------------------------------- |
| `offsetMs`   | `number \| null` | サーバーの NTP オフセット (ms)。未取得時は `null` |
| `lastSync`   | `string \| null` | 最終同期時刻 (ISO 8601)                           |
| `server`     | `string`         | 使用中の NTP サーバー名                           |
| `error`      | `string \| null` | NTP 問い合わせエラー                              |
| `serverTime` | `number`         | レスポンス生成時のサーバーの `Date.now()` (ms)    |

**`serverTime` の役割**: クライアントがサーバー時計のずれを考慮し、クライアント自身の時計に対する真の NTP オフセットを計算するために必要。

## Phase 3: クライアント ↔ サーバー

### ポーリング間隔

クライアントは初回ロード時および **30秒間隔** で `/api/ntp-offset` を取得する。

### 処理フロー

```
t1 = クライアントが HTTP リクエストを送信した時刻 (Date.now())
     ↓
   [HTTP リクエスト → サーバー処理 → HTTP レスポンス]
     ↓
t4 = クライアントが HTTP レスポンスを受信した時刻 (Date.now())
```

サーバーからのレスポンスに含まれる値:

```
serverTime = サーバーがレスポンスを生成した時刻
offsetMs   = サーバーの NTP オフセット (Phase 1 で算出)
```

### クライアント側オフセット計算

```
NTP時刻@サーバー = serverTime + offsetMs
RTT_client      = t4 − t1
clientMid       = t1 + RTT_client / 2
clientOffset    = NTP時刻@サーバー − clientMid
```

- `NTP時刻@サーバー`: サーバーがレスポンスを生成した瞬間の NTP 正確時刻
- `RTT_client`: クライアントとサーバー間の HTTP 往復時間
- `clientMid`: HTTP リクエストの中間時点におけるクライアントのローカル時刻
- `clientOffset`: **クライアントのPC時刻** に対する NTP 時刻のずれ（ミリ秒）

### NTP 補正済み時刻の取得

```javascript
function ntpNow() {
  return Date.now() + clientOffset;
}
```

アプリケーション内では `Date.now()` の代わりに `ntpNow()` を使用し、カウントダウン制御・時刻表示などすべてのタイミング処理に用いる。

## 全体のオフセット伝播図

```
NTP サーバー (正確な時刻源)
  │
  │  UDP NTP (RTT_server)
  ▼
サーバー時刻 ────── serverOffset = T_ntp − T_local_mid
  │
  │  HTTP (RTT_client)
  ▼
クライアント時刻 ── clientOffset = (serverTime + serverOffset) − clientMid
  │
  ▼
ntpNow() = Date.now() + clientOffset  ≈  NTP 正確時刻
```

## 精度に影響する要素

| 要素                    | 影響                                       | 対策                                    |
| ----------------------- | ------------------------------------------ | --------------------------------------- |
| NTP UDP RTT             | 非対称な遅延があると `serverOffset` に誤差 | 60秒ごとに再同期                        |
| HTTP RTT                | 非対称な遅延があると `clientOffset` に誤差 | 30秒ごとに再同期                        |
| サーバー処理時間        | `serverTime` 算出後のレスポンス送信遅延    | `serverTime` はレスポンス生成直前に取得 |
| JavaScript タイマー精度 | `Date.now()` の分解能 (通常 1ms)           | ±1ms の誤差は許容範囲                   |
| ネットワーク不安定      | RTT が大きいとき中間点の仮定が不正確になる | UI に RTT を表示し品質を可視化          |

## UI 表示

クライアントは画面上に以下を表示する:

- **NTP 補正済み時刻**: `ntpNow()` に基づく `HH:MM:SS.mmm` 形式
- **システム時刻**: 補正なしの `Date.now()` による `HH:MM:SS.mmm` 形式
- **NTP 補正情報**: クライアント基準オフセット、NTP サーバー名、サーバー側オフセット
- **HTTP RTT**: クライアント↔サーバー間の往復時間

### NTP 補正情報の表示フォーマット

```
NTP補正(クライアント基準): +12.3 ms (ntp.nict.jp, サーバー側offset +5.2ms)
HTTP RTT: 8 ms
```

- **クライアント基準オフセット**: `clientOffset` の値。1000ms 未満は `±X.X ms`、1000ms 以上は `±X.XXX s` で表示
- **サーバー側offset**: サーバーの `ntpOffset`（Phase 1 で算出した値）を併記し、問題切り分けに利用

### オフセット表示の色分け

| 条件                       | 表示色       | 意味       |
| -------------------------- | ------------ | ---------- |
| \|offset\| ≤ 100ms         | 標準（緑系） | 良好       |
| 100ms < \|offset\| ≤ 500ms | 警告（黄系） | 注意       |
| \|offset\| > 500ms         | 異常（赤系） | 大きなずれ |

### RTT 表示の色分け

| 条件                | 表示色       | 意味                 |
| ------------------- | ------------ | -------------------- |
| RTT ≤ 200ms         | 標準（緑系） | 良好                 |
| 200ms < RTT ≤ 500ms | 警告（黄系） | ネットワーク遅延注意 |
| RTT > 500ms         | 異常（赤系） | 精度低下の恐れ       |

### NTP 同期状態ごとの UI 表示

| 状態                | 表示内容                                                                                       |
| ------------------- | ---------------------------------------------------------------------------------------------- |
| 同期成功            | `NTP補正(クライアント基準): +X.X ms (ntp.nict.jp, サーバー側offset +Y.Yms)` + `HTTP RTT: Z ms` |
| サーバー NTP エラー | `サーバーNTP取得失敗 (エラーメッセージ)`                                                       |
| サーバー NTP 未同期 | `サーバーNTP未同期 (offsetMs=null)`                                                            |
| 通信エラー          | `通信エラー (エラーメッセージ)`                                                                |

同期失敗時は `ntpOffsetMs = null` となり、`ntpNow()` は補正なしの `Date.now()` をそのまま返す。

## 制約・前提

- NTP サーバーへの問い合わせはサーバー側のみが行い、ブラウザから直接 NTP (UDP) 通信はできない
- サーバーが 1 つの NTP サーバーのみに問い合わせるため、NTP サーバー自体の障害時はオフセットが更新されない（最後の成功値を使い続ける）
- HTTP 通信の RTT が極端に大きい環境（衛星通信等）では精度が低下する
- **実行基盤によっては UDP ポート 123 がブロックされ、NTP クエリが失敗する場合がある**（後述のトラブルシューティング参照）

## キャッシュ防止

HTTP レスポンスがプロキシやブラウザにキャッシュされると `serverTime` が古くなり、オフセット計算が不正確になる。以下の対策を実施:

- サーバー: `Cache-Control: no-store, no-cache, must-revalidate` ヘッダーを付与
- クライアント: `fetch()` に `{ cache: 'no-store' }` を指定し、クエリパラメータにタイムスタンプを付与

## トラブルシューティング

### 「NTP補正済時刻」と「システム時刻」がほぼ同じに見える

**正常なケースと異常なケースの2通りがある:**

| 状態                                  | UI 表示                                                                | 意味                         |
| ------------------------------------- | ---------------------------------------------------------------------- | ---------------------------- |
| NTP 同期成功 + クライアント時計が正確 | `NTP補正(クライアント基準): +2.3ms`                                    | 正常。補正量が小さいだけ     |
| NTP 同期失敗                          | `サーバーNTP取得失敗 (...)` または `サーバーNTP未同期 (offsetMs=null)` | 異常。補正が適用されていない |

### 二つの端末で NTP 補正済み時刻が一致する

**これは正常動作。** NTP 同期の目的は全端末が同じ正確な時刻を表示すること。端末AとBで `ntpNow()` が一致するのは、各端末ごとにクライアント基準のオフセットが正しく計算されている証拠。

確認方法:

- 端末Aと端末Bで「サーバー側offset」の値は同じ（サーバーの NTP オフセット）
- 端末Aと端末Bで「NTP補正(クライアント基準)」の値は**異なる**（各端末の時計のずれが反映される）
- 両端末のシステム時刻が同じなら、クライアント基準オフセットも類似になるのは正常

### 外向き UDP がブロックされた環境で NTP クエリが失敗する場合

クラウドの PaaS やコンテナ環境など、外向きの UDP 通信がブロックされている環境では UDP ポート 123 (NTP) への問い合わせが失敗する。症状:

- UI に `サーバーNTP取得失敗` と表示される
- `ntpNow()` がシステム時刻そのまま（補正なし）になる

対策:

- HTTP ベースの時刻取得サービスへの切り替えを検討（例: WorldTimeAPI）
- サーバーのホスト OS の時刻同期機能を利用（多くのクラウド VM はホストから高精度同期済み）
